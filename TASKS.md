## #5 â€” EventLoop::run dispatch loop

Implement `EventLoop::run(&mut self, handler: &mut dyn EventHandler) -> io::Result<()>` â€” the dispatch loop that fuses the timer wheel, `mio::Poll`, and the `EventHandler` trait into the runtime contract consumers see. Each iteration performs five steps in order. First, compute the poll timeout from `timer_wheel.next_deadline()`, falling back to a sane default cap (the wheel's 512 ms range) when no timer is pending so the loop never oversleeps a tick. Second, call `Poll::poll(events, timeout)`, propagating I/O errors except `ErrorKind::Interrupted`, which must be retried transparently â€” `EINTR` is a kernel artefact, not a consumer-visible error. Third, advance the timer wheel against `Instant::now()` and dispatch each returned `TimerId` through `handler.on_timer`; the wheel has already filtered cancelled ids in #2 so the loop does no extra bookkeeping. Fourth, iterate the polled `Events`, splitting the reserved waker token from consumer tokens â€” the waker dispatches at most one `handler.on_wake(registry)` per iteration regardless of how many concurrent `wake()` calls coalesced (this is the cost of `mio::Waker`'s level-triggered semantics, and ADR-001 Â§3 expects the consumer to coordinate via shared state, not via a wake-count), while every other token translates `mio::Event` readiness flags into `ReadyState` and dispatches `handler.on_event(registry, token, ready_state)`. Fifth, check `running.load(Ordering::Relaxed)` and return cleanly if `stop()` was called from a callback. The same `&Registry` is borrowed for the duration of every callback so consumers can register sources, change interest, schedule and cancel timers, all in-band â€” this is the immediate-modification semantics ADR-001 Â§1 mandates over command-queue alternatives. Tests should drive the loop with a recording handler and assert: a registered pipe firing the correct `ReadyState`; a timer scheduled via `Registry::insert_timer` firing exactly once; an external thread's `wake()` causing exactly one `on_wake`; a callback that calls `cancel_timer` before its target fires never receives `on_timer`; and a callback that flips an `AtomicBool` observed by the owner thread leading to `stop()` cleanly terminating the loop on the next iteration boundary. Depends on #4.

## #6 â€” rustikv integration: replace thread-per-connection with EventLoop

Migrate `rustikv`'s TCP server from the thread-per-connection model to `mio-runtime`'s `EventLoop`. Today each accepted connection spawns a dedicated OS thread that performs blocking reads against the BFFP framing parser; under load this caps connection scalability, inflates memory per idle peer, and makes connection limits awkward to enforce. The replacement runs a single `EventLoop` on the main thread, registers the listening `TcpListener` under `Token(0)` for `READABLE`, and on each `on_event` for that token loops `accept()` until `WouldBlock`, registering every accepted `TcpStream` with a fresh `Token` (allocated from a monotonic counter) and `READABLE` interest. Per-connection state moves into a `Connection` struct holding `read_buf: Vec<u8>`, `write_buf: Vec<u8>`, `write_offset: usize`, and a `ParseState` enum with two variants: `ReadingHeader` when fewer than the BFFP fixed-header bytes have been buffered, and `ReadingPayload { remaining: usize }` once the header is parsed and the body length is known. The handler owns a `HashMap<Token, Connection>` and on `on_event` reads into `read_buf` until `WouldBlock`, advances the `ParseState` as bytes accumulate, and dispatches each fully-framed request to the existing storage engine; on response, the encoded reply is appended to `write_buf` and the source is `reregister`ed for `READABLE | WRITABLE`. On the next `WRITABLE` event the connection drains `write_buf[write_offset..]` until `WouldBlock`, advancing `write_offset`, and reverts interest to `READABLE` only once the buffer is fully flushed. Connections are torn down on EOF, parse errors, or write errors by calling `Registry::deregister` followed by `HashMap::remove`. The runtime itself stays oblivious to TCP throughout â€” it sees only `&mut dyn mio::event::Source` â€” preserving the ADR-001 constraint that `mio-runtime` is not TCP-aware. Out of scope: replacing rustikv's compaction worker or other background subsystems with timers (the runtime exposes the primitives, but rewiring those subsystems belongs in follow-up tasks specific to rustikv); also out of scope is any change to BFFP wire format. Depends on #5.

## #7 â€” Investigate extending timer range beyond ADR-001's 512 ms

ADR-001 deliberately bounded the timer wheel at 512 slots Ă— 1 ms around the largest deadlines mio-runtime currently needs to host (Raft heartbeats and election timeouts), and #2 enforces that bound at runtime with an `assert!` on `insert(delay)` against the wheel's range. This task investigates what supporting longer delays would mean if a future consumer requirement â€” lease durations, snapshot timeouts, exponential-backoff retries, or anything else above the half-second mark â€” demands it. The deliverable is an architecture decision record (e.g. `docs/adr/ADR-002-extended-timer-range.md`), not code. The investigation must survey the four options sketched in the #2 conversation: (a) widening the flat wheel â€” same algorithm, linear memory growth, suitable when worst-case delay is bounded in the seconds-to-minutes range; (b) per-entry round counters â€” each slot stores `(TimerId, rounds_remaining)`, decremented on sweep and fired only at `rounds == 0` â€” the cleanest single-wheel extension at the cost of a more involved `next_deadline()`; (c) hierarchical timer wheels in the Linux-kernel pattern â€” multiple wheels at increasing granularities with cascading on rollover, O(1) amortised, effectively unbounded range, but with significant cancellation and `next_deadline()` complexity; and (d) overflow binary heap for long-tail timers â€” simpler than HTW but introduces two parallel data structures and two cancellation paths. The ADR must identify the triggering consumer requirement (no requirement â†’ no implementation), pick one option with a justified trade-off table, confirm the public `Registry::insert_timer(delay) -> TimerId` surface stays unchanged, and describe the test matrix that would gate implementation: cross-magnitude inserts, cascading correctness if applicable, cancellation across structures, and `next_deadline()` accuracy. Implementation itself is explicitly out of scope and becomes a follow-up task once the ADR lands. Depends on #2.

## #3 â€” Registry wrapping mio::Registry and the timer wheel

Define `Registry` as a thin wrapper around `mio::Registry` paired with a shared handle to the timer wheel from #2, exposing exactly the surface specified in ADR-001 â€” `register`, `reregister`, `deregister`, `insert_timer`, `cancel_timer` â€” and nothing more. The I/O methods are generic over `S: mio::event::Source` so the runtime stays decoupled from any particular I/O type, which is the constraint that lets the same runtime serve both rustikv (TCP) and raft-rs (TCP, possibly Unix sockets) without growing TCP-specific code paths. `Registry` must not be `Clone` and must not be `Send`: it is constructed internally by `EventLoop`, handed to callbacks by reference for the duration of a single dispatch, and never owned by the consumer â€” the type system enforces ADR-001's contract that consumers cannot smuggle a `Registry` out of a callback. Because the same `&Registry` is used to both schedule and cancel timers during a callback, the embedded timer-wheel handle must support interior mutability â€” wrapping the wheel in a `RefCell` (or equivalent single-threaded cell) is the natural shape and is consistent with the single-threaded design that ADR-001 commits to. Test in isolation by constructing a `Registry` from a real `mio::Poll` plus a stub timer wheel, registering a cross-platform `mio::event::Source` (e.g. `mio::net::TcpListener` bound to port 0), and asserting register/reregister/deregister forward correctly to the underlying `mio::Registry`, and that `insert_timer`/`cancel_timer` round-trip against the embedded wheel. Depends on #1 and #2.

## #2 â€” Hashed timer wheel

Implement a single-tier hashed timer wheel as a standalone module with no `mio` dependency, since timer scheduling is conceptually independent of I/O readiness and far easier to test in isolation than anything that touches `Poll`. The wheel has 512 slots of 1 ms each, giving a 512 ms total range â€” deliberately sized in ADR-001 around the largest deadlines mio-runtime needs to host: Raft election timeouts (150â€“300 ms) and heartbeats (50 ms), both comfortably inside one revolution, so multi-revolution scheduling is explicitly out of scope. The module must expose `insert(delay: Duration) -> TimerId` in O(1) by computing the target slot via modular arithmetic on the current cursor, appending to that slot's vector, and returning a fresh monotonically-increasing `TimerId`; `cancel(id: TimerId)` via lazy deletion â€” flipping a `cancelled` flag on the entry rather than scanning, so cancellation stays O(1) and never perturbs slot ordering; `advance(now: Instant) -> Vec<TimerId>` which walks the wheel from the previously-advanced tick up to the current tick, drains every slot it crosses, and filters out cancelled entries before returning; and `next_deadline() -> Option<Duration>` which tells `EventLoop::run` how long it can safely block in `Poll::poll` before the next timer is due, returning `None` when no timer is pending so the loop is free to block indefinitely. Tests must cover insertion across slot boundaries (including the cursor-wraparound case), lazy cancellation correctness, `advance` returning expired ids in chronological order, and `next_deadline` accuracy when the next timer is in the current slot, the next slot, and the slot just before wraparound. Depends on #1 for `TimerId`.
